## Abstract
Vision-Language-Action (VLA) models are emerging robotic control systems that integrate perception, language understanding, and action generation in a unified architecture. Existing testing approaches for VLA-enabled robots rely on manually constructed symbolic test oracles that determine task success from final environment states. These oracles are costly to construct, require domain expertise, and are often tightly coupled to specific tasks and environments, limiting scalability and reuse. Furthermore, they provide only end-state assessments of task outcomes, offering limited insight into intermediate behavior and fault localization. To address these limitations, we introduce MANGO, a multi-agent framework that automatically generates fine-grained oracles from natural-language descriptions of robotic tasks. MANGO first generates a reusable library of atomic tasks, then generates simulator-grounded oracle definitions for each atomic task, and finally produces executable fine-grained oracles by decomposing complex instructions into ordered sequences of atomic actions and corresponding oracles. The framework uses collaborative Generator, Assessor, and Judge agents that iteratively refine generated artifacts through structured feedback. We evaluate MANGO on the LIBERO_10 and RoboCasa Humanoid Tabletop benchmarks. Results show that MANGO generates executable, fine-grained oracles that detect a similar number of failures as symbolic oracles while accurately localizing them and providing richer diagnostic information. Through ablation studies, we further analyzed component contributions and the effect of initial task set, while preserving oracle quality. Overall, the results show the feasibility and effectiveness of test oracle generation for VLA-enabled robots testing.

## 1. Core Problem and Overall Meaning

Vision-Language-Action (VLA) models are a new class of robotic controllers that integrate visual perception, natural language understanding, and action generation. Testing these robots is currently done using **symbolic test oracles** that check only the final environment state against a goal (e.g., “Is the bottle inside the fridge?”). These oracles are expensive to create manually, tied to specific tasks and simulator setups, and provide only a binary pass/fail verdict. They cannot distinguish why a task failed (e.g., failure to open a door vs. failure to grasp an object) nor localize the failure to a specific step. As VLA robots perform long-horizon, multi-step tasks, the lack of fine-grained diagnostic information becomes a critical bottleneck for debugging and reliability.

The authors propose **fine-grained oracles** that decompose a complex instruction into a sequence of atomic tasks (e.g., `Open(Fridge)`, `Pick(Bottle)`, `Place(Bottle, Fridge)`), each with its own oracle. This allows structured verification of intermediate states and ordering constraints. The main challenge is generating such oracles automatically. **MANGO** addresses this by using a multi-agent LLM architecture to generate a reusable atomic task library, oracle definitions, and executable fine-grained plans from natural language instructions alone.

## 2. Methodology

### Core Idea
MANGO operates in three offline modules:
1. **Atomic Task Library Generation** – From a small initial set of complex task instructions, a **Generator** (World Model) produces a library of atomic tasks (e.g., `pick_up`, `place_in`, `open`, `close`). Each task is described with pre- and post-conditions. An **Assessment Board** (specialized Assessors) checks atomicity, redundancy, and repetition. A **Judge** (reasoning LLM) either approves the library or issues refinement instructions. This iterates up to 10 times.
2. **Atomic Task Oracle Library Generation** – For each atomic task, an **Generator** (World Model) creates a logical expression (e.g., `up(obj_id) and incontact(obj_id, robot_gripper)`) using simulator‑provided predicates (e.g., `in`, `on`, `turnon`). An **Oracle Hacker** (Assessor) identifies weaknesses, and a **Judge** verifies consistency across the whole library. This also iterates up to 10 times per atomic task.
3. **Fine-Grained Oracle Generation** – For each new complex instruction, a **Generator** produces a JSON plan that maps the instruction to an ordered sequence of atomic actions, each linked to its oracle via `handle_as`. An **Assessment Board** (Logic, Compliance, Grounding, and Execution Reliability Assessors) evaluates correctness. The **Judge** either approves the plan or provides specific fixes. This iterates up to 25 times.

### Key Technical Details
- **Agent Profiles**:  
  - **Generator** – World Model (e.g., Gemini Robotics ER) for physical reasoning.  
  - **Assessor** – Lightweight LLM for fast, parallel checks (atomicity, redundancy, grounding, etc.).  
  - **Judge** – Reasoning LLM (e.g., Gemini Robotics ER) that synthesizes all assessments and issues structured refinement instructions.  
- **Output**: A fine-grained oracle is a JSON structure containing `instruction`, `execution_mode` (sequential or `any_order`), and a `plan` with steps, each step having actions and monitoring oracles.  
- **Refinement Process**: Each module uses iterative loops where Generator candidates are judged; if rejected, specific instructions guide next generation.

- **RQ1 (Generation Capability)**: MANGO achieves 0.79–0.93 executability and 0.91–0.95 similarity, significantly outperforming baselines (which often produce 0% executability). MANGO is slower than single-pass baselines (offline use), but much faster than MANGO_o due to better convergence.

- **RQ3 (Ablation)**: Removing any component significantly reduces executability and similarity (e.g., w/o Module I: executability drops to 0.22 on RoboCasa; w/o Execution Reliability: executability < 0.05). Each module’s contribution is critical.

## 7. Strengths

- **Automation**: First fully automated end-to-end method for generating executable fine-grained oracles for VLA robots from natural language alone.  
- **Reusability**: Atomic task and oracle libraries are generated once per benchmark and can be reused for any new instruction, drastically reducing manual effort.  
- **Diagnostic Power**: Provides step-level failure localization, enabling debugging that symbolic oracles cannot.  
- **Rigorous Evaluation**: Multiple benchmarks, statistical tests, ablation studies, and a minimization analysis – all with replication across 10 runs.  
- **Practical Impact**: Shows that a small, diverse set of initial tasks is sufficient, which is highly relevant for industrial adoption.

## 8. Limitations

- **Dependence on Simulator Fidelity**: The quality of generated oracles is constrained by the granularity and correctness of available simulator functions (e.g., `is_held`, `inside`). RoboCasa’s less reliable functions caused higher false positives/negatives.  
- **Generation Overhead**: While offline, MANGO is significantly slower than single-pass baselines (up to 62× in Module III), which could limit iteration speed.  
- **Scope of Evaluation**: Only two table-top manipulation benchmarks (household scenarios). Generalization to industrial settings, real-world robots, or domains with high sensor noise is untested.  
- **Stochasticity**: Foundation model outputs vary; the iterative refinement mitigates this but does not eliminate risk of occasional invalid oracles.  
- **Human Validation Still Needed**: Some false positives/negatives required manual inspection; in safety-critical contexts, automated oracles may need additional verification.
